# Re-Review Procedure (Phase 9)

Heavy procedure for re-reviewing one change-requested PR. Invoked per-PR by the sweep, or directly ("re-review #N"). Mirrors the lead maintainer's review skeleton; upgrades it with exogenous verification, an ADR/source-of-truth pass, and a cross-reference pass.

**Verdict standard (Google eng-practices):** clear it once it *definitely improves code health*, even if imperfect — nits do **not** block. A maintainer MAY elevate a minor to required, but only with a stated reason in the comment. Small nits stay as non-blocking polish.

## Steps (in order)

**Repo-specific steps:** 3a (generated-source / ADR-457), 6a (capability-vs-core / ADR-857+1244), and 6b (changeset purity) encode *gsd-core* invariants. On another repo, substitute that repo's equivalent invariants or skip them — the rest of the procedure is repo-agnostic.

1. **Re-fetch live state + treadmill guard.** PR HEAD OID, reviews, labels, CI check-run conclusions, mergeStateStatus. Everything below is against this snapshot; if HEAD moves, restart. **If required checks are still in-progress, CI evidence is incomplete — wait for completion (or mark CI `pending`); never finalize a `CLEAR`/merge on partial CI.** **Stop condition:** count prior AI re-review rounds on this PR — if you've already posted ≥2 and HEAD has no *new substantive* change since the last (only rebases/merges), do NOT post again; escalate to a human maintainer. Re-reviewing on every push triggers dismiss-stale-on-push and an infinite loop.
2. **Gather.** PR diff; linked issue + its brief; the latest `CHANGES_REQUESTED` review + its inline threads; rebase/conflict state.
3. **Merge-no-op / revert detection** (the #1418 catch). `git diff origin/<base>...HEAD -- <changed files>` — confirm the PR still changes what it claims. A merge that resolved conflicts by taking the base side can silently revert the whole PR → HEAD becomes a no-op. If empty/reverted → **Blocker**, stop.
3a. **Generated-vs-source check (ADR-457).** `bin/lib/*.cjs` are GENERATED from `src/*.cts`. If the diff edits a generated `bin/lib/*.cjs` directly, or a test requires the build artifact instead of the live `bin/install.js`/source path (the #1418 blocker shape), that's a **Blocker** — the contributor must edit the `src/*.cts` source of truth and run `build:lib`. Confirm the source file changed and the generated output matches a fresh `build:lib` (no stale or hand-edited artifact). **Hand-synced pairs:** some functions are duplicated across `src/*.cts` and a hand-written `bin/install.js` (the CJS↔SDK seam; `lint-shared-module-handsync` guards drift) — e.g. `convertGeminiToolName` lives in both. When the diff touches one copy, verify the other is updated in lockstep; a one-sided edit is a **Blocker**. **Registration surfaces (`ci-preflight`):** a newly *added* shipped file/command/export must appear in every registry its siblings do — grep a sibling across the repo (command-alias arrays, skill `requires:` closure, `CONTEXT.md`, inventory/drift-guard tests). Missing registration passes unit tests but fails integration/drift-guard CI — recognize that failure class in 4a.
4. **Local verification — actually run it.** `npm ci`, `build:lib`, `lint:ci`, `test:unit` (catches drift-guard tests `npm test` can miss), the named test files for the touched module. Capture real output and counts. If a command can't run, write "not run" — **never** fabricate a number. **Fork PRs:** a fork head branch isn't fetchable from `origin` by default, so local runs usually aren't possible without `gh pr checkout` (which disrupts the working tree) — mark local "not run (fork PR)" and treat **CI conclusions as the evidence**. **Cross-platform caveat:** local runs on one OS/Node version and cannot catch platform-specific failures (#1400 Windows, node24-only). For diffs touching install/path/process/hook code, the **CI matrix (ubuntu/macOS/Windows, node22/24) is the authority** — local green is necessary, not sufficient.
4a. **CI attribution — read the ACTUAL failing assertion FIRST, then attribute.** A failing required check is a merge-blocker but not necessarily a *code* defect — but you MUST find the real cause before judging.
   - (a) **Grep the failed-step log for the literal failure line** (`✖` / `not ok` / `AssertionError` / `##[error]`). Do NOT infer the cause from nearby output — a coverage table printed just before `exit 1` does NOT mean coverage failed; a `✖ product name purity` higher up is the real cause. (This exact trap produced a wrong APPROVE on #1409: a changeset-prose `product-name-purity` failure misread as a "coverage/stale-base artifact.")
   - (b) Once you have the **named** failing test/assertion, attribute it: is it the PR's own change (its changeset prose, its code) or pre-existing?
   - (c) Only call something environmental/stale-base after confirming **`compare ... behind_by > 0`** AND that `next` + siblings fail the **same named test** — never from `mergeStateStatus: BEHIND` alone (it can read BEHIND while `behind_by=0`), and never by matching two *different* reds. Record the named failure as evidence; never assert "flaky"/"stale base" without it.
4b. **Fail-first regression test (bug-fix PRs) — `test-first-bugfix` discipline.** A fix without a reproduction test is a guess. For a bug-fix PR, verify (a) it ADDS a regression test for the reported defect, and (b) the test is *proven fail-first* — it fails on the pre-fix code for the bug's reason, not merely passes now. A fix with no test, or a test that would pass without the fix, is a **Blocker** (a Nit on a trivial fix). Core gsd-core value (machine-fail-first, #1279).
5. **Map prior findings.** One row per earlier finding → `Resolved / Partial / Open / Contributor-declined / Maintainer-action`, each with commit-SHA or `file:line` evidence. Distinguish a code defect from a maintainer-only action (e.g. an external contributor can't self-apply a label).
6. **ADR & source-of-truth pass.** Identify the ADR(s) governing the touched seams: grep `docs/adr/` and `CONTEXT.md` for the changed module/function names. Verify the diff **UPHOLDS** each governing decision (don't just confirm an ADR exists). Then check the inverse: does the change add/alter an exported seam, predicate, or behavior that `CONTEXT.md` or an ADR must record but doesn't? A missing glossary update is a **Blocker even when CI is green** (drifts the canonical spec).
6a. **Capability-vs-core boundary verdict (ADR-857 / ADR-1244).** This is the lead maintainer's primary architectural mandate — render it explicitly: **UPHELD / BREACHED**. Check (a) **host-vs-capability boundary** — core dispatch (hardcoded `case` in `gsd-tools.cjs`) must run *before* the `default:` capability-dispatch path; new behavior that belongs in a capability must not be hardcoded into core, and vice-versa; (b) **trust model** — repo-scoped ledger = consent, no env/cwd bypass, ledgerless overlays don't auto-activate, never-crash `loadRegistry` invariant; (c) **integrity/confinement** — `--integrity` honored for npm/git, hook commands not written as unconfined relative paths. If the diff touches capability or core-dispatch code and you cannot confirm UPHELD, it is a **Blocker**. (See open capability bugs #1459/#1460/#1461 for the live failure shapes.)
6b. **Changeset + docs policy (fires when user-facing).** Verify a `.changeset/*.md` fragment exists; check its type — `Added`/`Changed`/`Removed` MUST touch `docs/` (`lint:docs`), and any `docs-exempt` marker needs a genuine rationale (the #1421 Blocker 2 shape: a behavioral change is user-visible even if the syntax is unchanged). `no-changelog` is valid only for docs-only / changelog-direct PRs. A missing or mis-typed changeset, or an unjustified `docs-exempt`, is a **Blocker**. Changeset *prose* also has content invariants enforced by node tests (not just `changeset-lint`) — e.g. `product-name-purity` forbids parenthetical product descriptions, since fragment text renders verbatim into CHANGELOG.md (the #1418 blocker). A purity-test failure is a **Blocker** even when `changeset-lint` is green.
7. **Cross-reference in-flight work.** Search other OPEN PRs and issues touching the same files / module / ADR / linked refs. Flag: **overlap/conflict** (another PR edits the same lines → rebase-order or mutual-break risk), **duplication/supersession** (an issue or PR already covers this defect), **dependency** (blocks or blocked by another item). Cite the `#numbers`.
8. **Scoped code review (defect-class bounded).** Run `/code-review` on the delta when the branch is local; **for a fork PR you can't fetch (step 4), review the diff via `gh pr diff <n>` instead** (the exogenous subagent in step 10 can also pull it independently). **Scope-frozen to the defect class** (per the #1462 playbook — do not let a small fix draw a sprawling audit). Report only findings on the change itself. **Security routing:** if the PR/issue is `security`-labeled or touches the capability trust model (ADR-857/1244 — ledger, integrity, confinement), a generic review under-serves — additionally run `/security-review` (or dispatch `gsd-security-auditor`) and treat its findings as blocking.
9. **Software-law lens** via `/skills-from-the-artificer`. Include a law ONLY when it maps to a concrete diff fact (e.g. Hyrum's on an observable-behavior change). Omit otherwise — no plausible-but-irrelevant citations.
10. **Exogenous self-check.** **REQUIRED before any `CLEAR`/Approve verdict** — and whenever a "resolved" call rests on *your own reading of the code* rather than a machine fact: dispatch a *fresh* reviewer subagent (e.g. `feature-dev:code-reviewer`) given only the diff + the blocking findings + the governing ADRs — **never your verdict** — and have it independently judge whether each blocking finding is resolved. Compare to your map; any discrepancy → do not post a clear verdict, investigate first. An agent ratifying its own conclusion is the failure mode, not the safeguard. **Optional** when the verdict is Request-changes AND the blocker is hard machine ground-truth (a CI assertion / failing-check log that names the cause): self-bias risks over-*approving*, not over-blocking, so a ground-truthed block carries no self-bias risk — record the CI evidence instead. **Then `trust-but-verify` the reviewer itself** (it is the audit subject, not the validator): treat its report as a *lead* and confirm each key "resolved"/"defect" claim against the **primary source** — the actual diff line or the CI assertion, quoted — never accept agent confidence as evidence.
11. **Verdict.** `CLEAR` (no open blockers; nits may remain) · `N BLOCKER(S) OPEN` · `CHANGES REQUESTED`. When the *code* is clear but merge is gated by non-code reasons (rebase, a CI artifact, or an original reviewer's *unverifiable* still-open CR — a verifiably-resolved CR is dismissed per 12a, not merge-blocking), use `CLEAR · merge-blocked: <reasons>` — it distinguishes "no code changes needed" from "ready to merge."
12. **Deliver as a single formal review** (the hard template below is the review body). Show it to the human; **submit only on their OK**, human-submitted — never a bot auto-approval, and do NOT also post a standalone comment (a standalone comment is only for interim status when you're *not* submitting a review). Map verdict → review event: `CLEAR` (incl. `CLEAR · merge-blocked`) → **Approve**; `N BLOCKER(S) OPEN` / `CHANGES REQUESTED` → **Request changes**. Submitted reviews are immutable in state and undeletable — pick the right event the first time.
12a. **Dismiss a stale change-request whose conditions are met.** If your verdict is `CLEAR` and another maintainer's `CHANGES_REQUESTED` is still open but the contributor has *verifiably resolved every change that reviewer requested* (all their findings → `Resolved` in step 5, confirmed by the exogenous reviewer in step 10), **dismiss that review** so the PR isn't stuck awaiting a manual re-flip: `gh api -X PUT /repos/<owner>/<repo>/pulls/<n>/reviews/<review_id>/dismissals -f message='…' -f event=DISMISS`. The dismissal message must (a) carry the AI disclaimer, (b) name each resolved requested change with its evidence (commit SHA / `file:line`), and (c) reference your re-review. **Guardrails:** dismiss ONLY when fully resolved AND verified — never to bypass an unmet or subjective request, and never if any blocker (theirs or newly found) remains. For the **lead maintainer's** CR on their own architectural line, `@`-mention them in the dismissal as a heads-up rather than dismissing silently. Dismissing fulfils their stated conditions; it does not override their judgment. **Like a merge, dismissing another maintainer's review is a consequential, socially-sensitive, hard-to-reverse write — do it ONLY on explicit human authorization, never autonomously.**
12b. **Update labels** (per **[labels.md](labels.md)**; re-fetch current labels first). Set the PR's review-state label (`review: approved` / `review: changes requested` (+`needs changes`) / `review: approved (merge conflict)` + `needs rebase`). Remove the superseded review-state label — **including a stale `review: changes requested` + `needs changes` once a prior reviewer's changes are confirmed addressed (all `Resolved`, no new blocker), even if you're not applying a new review-state label**. Apply any missing maintainer-one-click label mirrored from the issue (e.g. `approved-enhancement`). **Another maintainer's CR still open:** if verifiably resolved → it's dismissed (12a) → `review: approved`; if genuinely unmet/unverifiable → leave the formal state to them, don't mark `review: approved`, drop only a plainly-stale label (e.g. `needs-review`), note it (the sole `CLEAR · merge-blocked` case).
13. **Merge gate.** **Re-fetch HEAD OID, reviews, and CI conclusions immediately before merging** — long analysis (test runs, the exogenous subagent) may have gone stale; if anything changed since step 1, abort and restart. Merge ONLY if ALL hold: invocation carried `merge=#<n>` · verdict is plain `CLEAR` (NOT `CLEAR · merge-blocked`) · CI real conclusions green · rebased (not BEHIND/DIRTY) · review threads resolved · **no other maintainer has an *unresolved* open `CHANGES_REQUESTED`** (a verifiably-resolved one is dismissed per 12a, not merged-over). Otherwise print the exact merge command and stop ("merge authority: HELD").
14. **Post-rebase handoff (for `CLEAR · merge-blocked: rebase`).** Once the contributor rebases and CI goes green, the only thing left is the lead's merge. Switch the signal: apply `approved to merge` (per [labels.md](labels.md)) and ping the original reviewer to re-flip + merge. This is the state transition once the non-code blocker clears — don't leave an approved, rebased PR sitting silently.

## Comment template (HARD — fixed sections, fixed order, size-bounded)

**Contract — the posted body IS exactly the block below, no more:**
- **One review body per round, ≤ ~200 words visible.** All verification depth lives in the single `<details>` fold. Never post a second comment in the same round.
- **Suppress-empty.** Include an optional line ONLY when its `[only if …]` predicate holds. A clean optional check is *omitted*, not printed as "none" — the **Merge-readiness** line is the one that always prints.
- **Don't restate resolved findings each round.** On a re-review the Prior-findings table shows status only; the visible body focuses on what's still open.
- **Verbatim structure.** Do not add, reorder, or rename sections. Findings use trek-e's prose (**Blocker** / **Nit**) carrying explicit **(non-blocking)** / **(blocking)** semantics.

```
> *Generated by AI during triage; reviewed and submitted by a human maintainer (@<you>).*

## Re-Review — PR #<n> · **<CLEAR | CLEAR · merge-blocked: <reasons> | <N> BLOCKER(S) OPEN | CHANGES REQUESTED>**
Track: <bug|enh|feat> · issue #<m> `<gate-label>` (<satisfied|MISSING>)

**Prior findings** (from @<reviewer>, <date>)
| # | Finding | Sev | Status | Evidence |
|---|---|---|---|---|
| 1 | <one line> | Blocker/Nit | Resolved/Partial/Open/Declined/Maintainer | <sha> |

**Merge-readiness:** rebase=<…> · CI=<…> · threads=<…> · other-maintainer CR=<…> · **merge: HELD**
**Praise:** <one specific line>
[only if open findings exist:]
- **Blocker** — <…>
- **Nit** (non-blocking) — <…>   [or **Nit** (blocking) — elevated: <reason>]

<details><summary>Verification & checks</summary>

- Local: `npm ci` <…> · `lint:ci` <…> · `<test>` <n/n>  (<"not run" where applicable — never fabricated>)
- ADR-857/1244 cap-vs-core: <UPHELD/N/A> · ADR-457 source: <ok/BLOCKER> · CONTEXT.md: <sync/UPDATE> · changeset+docs: <ok/BLOCKER>
- Cross-ref: <none | #x overlap / #y supersedes / blocked-by #w>
- Independent reviewer: <agreed | DISCREPANCY on #k → verdict downgraded>
- Software-law: <Law: one line>   [only if it maps to a diff fact]
</details>
```

## Posting discipline

- The re-review is delivered **as** the formal GitHub **review** (Approve / Request-changes), not as a separate comment. Submit it when the verdict lands or changes; for interim status only, edit a single running triage comment rather than stacking new ones — keeps the timeline clean and avoids the CODEOWNERS-firehose.
- The collapsed `<details>` keeps the visible timeline short while preserving the full evidence trail for anyone who expands it.
- If the body would exceed the budget, cut from the `<details>`, never from the verdict / prior-findings / merge-readiness core.

## Red flags — STOP

- About to report a test count you didn't run → run it or write "not run."
- About to post "Clear" without the exogenous reviewer agreeing → don't.
- About to merge over another maintainer's *unresolved* change-request → don't, even with the token (a verifiably-resolved one is *dismissed* on human OK per 12a, not merged-over).
- About to dismiss another maintainer's review autonomously → don't; dismissal needs explicit human authorization (step 12a).
- Skipped the merge-no-op `git diff` → a reverted PR looks "addressed" but ships nothing.
- Skipped the ADR/CONTEXT inverse check → a green-CI PR can still drift the spec.
- Posting a long multi-section wall → use the hard template; depth goes in the one `<details>`, core stays ≤ ~200 words.
